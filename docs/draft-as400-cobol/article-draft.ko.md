# AS400 / COBOL Migration

AS/400(IBM i)와 COBOL/RPG로 동작하는 기간계 시스템을 AI로 분석하고, Java, PostgreSQL, Linux, Cloud 환경으로 단계적으로 이전하는 현대화 서비스입니다.

오랫동안 운영되어 온 기간계 시스템에는 수발주, 재고, 회계, 결제, 급여, 고객관리 등 기업 업무를 멈출 수 없는 중요한 로직이 축적되어 있습니다.

반면 유지보수 비용 증가, COBOL/RPG 엔지니어 고령화, 시스템 블랙박스화, 클라우드 전환 지연은 많은 기업에게 더 이상 무시하기 어려운 과제가 되고 있습니다.

QueryPie는 기존 AS/400(IBM i)와 COBOL/RPG 자산을 단순히 교체하는 것이 아니라, AI를 활용해 현행 시스템을 분석하고, 이전 범위, 영향도, 설계, 테스트, 구현 방침을 정리하면서 현실적인 현대화를 지원합니다.

## 일본 기업에 남아 있는 AS/400(IBM i)와 COBOL/RPG

일본에서는 AS/400(IBM i)나 COBOL/RPG로 구축된 기간계 시스템이 현재도 많은 기업에서 사용되고 있습니다.

특히 제조, 유통, 물류, 금융, 보험, BPO, 급여 계산처럼 멈출 수 없는 업무를 지탱하는 시스템에서는 오랫동안 운영되어 온 AS/400(IBM i) 환경이 지금도 중요한 역할을 맡고 있습니다.

이러한 시스템은 단순히 오래되었기 때문에 남아 있는 것이 아닙니다. 장기간 안정적으로 동작했고, 업무에 깊이 통합되어 있으며, 기업별로 복잡한 비즈니스 로직을 지탱해 왔기 때문에 쉽게 교체할 수 없습니다.

많은 기업에서는 다음과 같은 판단이 오랫동안 이어져 왔습니다.

> 동작하고 있으니, 지금은 건드리지 않는다.

하지만 이 판단을 계속 유지하기만 해서는 앞으로의 유지가 점점 어려워지고 있습니다.

## 왜 지금 AS400 / COBOL Modernization이 필요한가

AS/400(IBM i)와 COBOL/RPG의 문제는 시스템이 동작하지 않는다는 것이 아닙니다. 오히려 안정적으로 동작하고 있기 때문에 바꾸는 판단이 늦어지기 쉽다는 데 있습니다.

반면 운영 현장에서는 다음 문제가 커지고 있습니다.

- COBOL/RPG를 이해할 수 있는 엔지니어가 줄어들고 있음
- 기존 벤더나 특정 유지보수 담당자에 대한 의존도가 높아지고 있음
- 하드웨어, 라이선스, 유지보수 비용이 무거워지고 있음
- 오랜 기간의 수정으로 업무 로직의 전체상이 보이기 어려워지고 있음
- 영향 범위를 알기 어려워 개보수나 이전 판단이 늦어짐
- 클라우드, API, 데이터 활용 같은 DX 과제와 연결하기 어려움

일본에서는 경제산업성이 지적한 "2025년의 절벽"을 배경으로, 레거시 시스템의 블랙박스화, IT 인력 부족, 유지 비용 증가가 큰 경영 과제로 인식되고 있습니다.

AS400 / COBOL Modernization은 단순한 기술 쇄신이 아닙니다. 기업의 핵심 업무를 지탱하는 기존 자산을 살리면서, 다음 10년도 운영 가능한 형태로 바꾸어 가는 작업입니다.

## 전면 재개발이 아니라, 먼저 AI로 현행 시스템을 이해한다

AS/400(IBM i)나 COBOL/RPG의 이전이 어려운 이유는 코드 양만이 아닙니다.

정말 어려운 것은 오랜 운영 속에서 축적된 업무 로직, 데이터 구조, 외부 연계, 예외 처리, 배치 처리의 관계를 정확히 이해하는 것입니다.

QueryPie는 AI를 활용해 기존 시스템을 이해하는 것에서 시작합니다.

### AI 분석

- COBOL 프로그램 구조 분석
- RPG 로직 분석
- DB2 / Oracle 등 데이터 구조 분석
- 프로그램, 테이블, 배치, 외부 연계의 의존관계 정리
- 개수 또는 이전 시 영향도 분석

### AI 문서 생성

- 현행 사양서 생성
- 업무 로직 정리
- 데이터 모델 작성
- 이전 대상 분류
- 테스트 케이스 생성

이를 통해 먼저 "무엇을 이전해야 하는가", "어디서부터 착수해야 하는가", "어느 범위에 영향이 있는가"를 가시화합니다.

## Java / PostgreSQL / Linux / Cloud로 단계적으로 이전

현행 시스템을 이해한 뒤, 이전 대상 아키텍처를 단계적으로 설계합니다.

이전 방향은 기업 상황에 따라 달라질 수 있습니다.

- COBOL에서 Java로 변환
- RPG에서 Java로 변환
- DB2 / Oracle에서 PostgreSQL로 이전
- AS/400(IBM i)에서 Linux 환경으로 이전
- OCI / AWS 같은 클라우드 기반으로 이전
- 기존 기능을 API Wrapper로 단계적으로 외부화
- 기존 IBM i를 남기면서 일부 기능만 먼저 분리

중요한 것은 처음부터 전체를 교체하는 것이 아닙니다.

현행 시스템을 분석하고, 영향도를 파악하고, 업무상 우선순위를 정한 뒤, 작게 검증하면서 단계적으로 이전하는 것입니다.

### 1. 현황 진단

대상 시스템, COBOL/RPG 프로그램, 데이터베이스, 배치, 외부 연계를 확인하고 이전의 전체상을 파악합니다.

### 2. AI 분석

AI를 활용해 코드 구조, 업무 로직, 데이터 모델, 의존관계, 영향 범위를 분석합니다.
