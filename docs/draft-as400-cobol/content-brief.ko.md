# AS400 / COBOL Migration 콘텐츠 기획안

## 1. 콘텐츠 개요

### 작업명

일본 웹사이트용 IBM i（AS/400）Modernization 정적 웹페이지

### 콘텐츠 목적

일본 기업에 여전히 남아 있는 IBM i（AS/400）와 COBOL/RPG 기반 기간계 시스템을 대상으로, QueryPie AI가 제공할 수 있는 AI 기반 현대화 접근을 소개한다.

최신 출처 조사 기준으로 콘텐츠의 중심 메시지는 `낡은 시스템의 교체`가 아니라 `핵심 업무 자산의 안정적 현대화와 인력 리스크 해소`로 잡는다.

다음 문제를 주요 과제로 다룬다.

- IBM i（AS/400）와 COBOL/RPG 기반 기간계 시스템의 장기 운영 부담
- COBOL/RPG 엔지니어 고령화와 기술 승계 리스크
- 수십 년 누적된 비즈니스 로직의 블랙박스화
- 기존 벤더/SI 또는 특정 담당자 의존도 심화
- 레거시 시스템으로 인한 데이터 활용, API 연계, 클라우드 전환, DX 성과 창출 지연
- Java, API, PostgreSQL, Linux, OCI/AWS 기반 단계적 현대화 필요

### 게시 형태 우선순위

1차 게시 형태는 일본 웹사이트의 정적 웹페이지다.
공식 주 표기는 `IBM i（AS/400）モダナイゼーション`으로 사용하고, `AS400 / COBOL マイグレーション`은 SEO와 문맥상 보조 표기로 사용한다.

초기 구현은 `/services/as400-cobol` 직접 URL로 접근 가능한 콘텐츠 초안이다.
콘텐츠가 리뷰와 수정을 여러 번 거칠 예정이므로, 메뉴와 sitemap에는 아직 노출하지 않고 robots는 `noindex, nofollow`로 둔다.
한국어 확인용 페이지는 별도 공개 경로를 만들지 않고 `/services/as400-cobol?lang=ko` 쿼리 파라미터로 제공한다.
한국어 카피와 섹션 구성은 `src/app/services/as400-cobol/page.ko.tsx`가 route-local authoring 방식으로 소유한다.

후순위 확장 콘텐츠는 다음과 같다.

| 우선순위 | 형태 | 목적 |
| --- | --- | --- |
| 1 | 정적 웹페이지 | `/services/as400-cobol` 콘텐츠 초안 구현 |
| 2 | White paper | 리드 수집, 상세 진단 프레임, 세일즈 후속 자료 |
| 3 | Blog | 시장 교육, SEO 유입, 문제 인식 확산 |
| 검토용 | 직접 URL | 메뉴/sitemap 미노출 상태에서 stage 또는 production 확인. 한국어 확인은 `?lang=ko` 사용 |

## 2. 타깃 독자

### 주요 독자

- 일본 중견/대기업의 CIO, CTO, DX 추진 책임자
- 기간계 시스템, Core System, 基幹システム 담당 조직
- AS/400（IBM i）, COBOL, RPG 기반 시스템을 운영하는 정보시스템 부서
- 제조, 유통, 물류, 금융, 보험, BPO, Payroll 영역의 시스템 책임자
- 기존 SI/벤더 유지보수 비용과 전환 리스크를 동시에 고민하는 의사결정자

### 독자의 문제

- AS/400（IBM i）와 COBOL/RPG 시스템이 핵심 업무를 지탱하고 있어 쉽게 건드릴 수 없다.
- 시스템은 안정적으로 동작하지만 유지보수 인력, 지식 승계, 비용 문제가 커지고 있다.
- 수십 년 누적된 업무 로직이 코드 안에 남아 있어 전환 범위와 영향도를 파악하기 어렵다.
- 기존 벤더에게 높은 비용을 지불하지만, 구체적인 현대화 로드맵을 잡기 어렵다.
- 전체 재개발은 비용과 리스크가 크기 때문에 단계적이고 검증 가능한 전환 방식이 필요하다.
- 경영진에게 `왜 지금 시작해야 하는가`를 설명할 수 있는 객관적 근거가 필요하다.

## 3. 핵심 메시지

### 메인 메시지

IBM i（AS/400）와 COBOL/RPG 기반 기간계 시스템은 일본 기업의 핵심 업무를 지탱해 왔다.
그러나 이제는 유지보수 비용, 인력 부족, 블랙박스화, 데이터/API/클라우드 활용 제약을 해결하기 위한 단계적 현대화가 필요하다.
QueryPie AI는 AI를 활용해 기존 COBOL/RPG 로직과 데이터 구조를 분석하고, Java / API / PostgreSQL / Linux / OCI / AWS 기반 전환을 검증 가능한 단위로 추진할 수 있도록 지원한다.

### 보조 메시지

1. 레거시 시스템 현대화는 전체 재개발이 아니라, 먼저 현재 시스템을 정확히 이해하는 것에서 시작한다.
2. IBM i는 여전히 핵심 업무 플랫폼이므로, 안정성을 해치지 않는 단계적 접근이 필요하다.
3. AI는 COBOL/RPG, DB2/Oracle, 영향도 분석, 설계서 생성, 테스트 케이스 생성에 활용될 수 있다.
4. 분석 결과를 바탕으로 Java 변환, API 생성/연계, PostgreSQL/Linux/OCI/AWS/Cloud 전환을 단계적으로 검토한다.
5. 핵심 업무를 멈추지 않기 위해 진단, PoC, 변환, 구현, 테스트, 병행 가동, 운영 안정화까지 이어지는 진행 모델이 필요하다.
6. 일본 시장에서는 METI `2025年の崖`, IPA `DX動向2025`, COBOL 기술자 부족, IBM i 핵심 업무 의존도가 현대화 수요의 근거가 된다.

### 섹션 중복 방지 원칙

- `Service Scope`, `Migration Flow`, `Scenario`를 별도 섹션으로 과도하게 나누지 않는다.
- 단계적 진행 모델과 실제 업무 시나리오는 `Service Scope` 카드 안에 녹인다.
- `Service Scope`는 단순 기능 목록이 아니라 “어떤 업무 단위에서 무엇을 확인하고, 어떤 순서로 위험을 줄이며, 어디까지 구현/운영을 지원하는가”를 보여주는 섹션이다.
- 별도의 운영 시나리오 섹션을 둘 경우 앞선 서비스 범위와 같은 내용을 반복하기 쉬우므로, 수발주, 재고, 청구, 배치, 장표, 외부 연계 같은 예시는 카드 본문 안에서 사용한다.
