# 게시 전 체크리스트

## 1. 솔루션 메시지 검증

- [x] `AS400 / COBOL Migration`을 QueryPie의 솔루션 또는 서비스로 명시 가능
- [ ] `IBM i（AS/400）モダナイゼーション`과 `AS400 / COBOL マイグレーション` 중 주 표기 결정
- [x] AI 기반 COBOL/RPG 분석, 설계서 생성, 테스트 케이스 생성, Java/API/Cloud 전환은 충분히 공개 가능
- [ ] DB2 / Oracle 분석, PostgreSQL 이전, Linux/OCI/AWS 이전을 구체적으로 언급해도 되는지 확인
- [x] Payroll 사례의 구체 일정 표현은 일반 홈페이지 공개 문구에서 사용하지 않음

## 2. 일본어 품질 검수

- [ ] 일본어 원어민 또는 일본 마케팅 담당자 검수
- [ ] `AS/400`, `IBM i`, `COBOL`, `RPG`, `レガシーシステム`, `基幹システム`, `モダナイゼーション` 표기 통일
- [ ] 과도하게 한국식 영업 표현으로 보이는 문장 제거
- [ ] 일본 B2B 기술 서비스 페이지 톤에 맞게 문장 길이와 단락 조정
- [ ] `AS400`과 `AS/400`의 표기 정책 결정

## 3. SEO 검수

- [ ] H1 또는 메타 타이틀에 `AS400 / COBOL Migration` 또는 `IBM i（AS/400）モダナイゼーション` 포함
- [ ] H2/H3에 `COBOL マイグレーション`, `RPG`, `基幹システム`, `レガシーシステム刷新`, `2025年の崖` 등 관련 키워드 포함
- [ ] 메타 타이틀 작성
- [ ] 메타 디스크립션 작성
- [ ] 일본 웹사이트 내부 관련 페이지로 링크 추가
- [ ] 문의/데모/진단 CTA 링크 추가

## 4. 게시 위치 결정

- [x] 1차 게시 형태는 일본 웹사이트 정적 웹페이지로 결정
- [ ] 정적 웹페이지가 들어갈 솔루션/서비스 섹션 결정
- [ ] White paper 확장 여부는 후순위로 검토
- [ ] Blog 확장 여부는 후순위로 검토
- [ ] `corp-web-japan`의 콘텐츠 구조 확인
- [ ] 정적 페이지 구현 방식 확인
- [ ] 슬러그 결정
- [ ] canonical URL 정책 확인

## 5. 근거와 표현 리스크

- [ ] 일본 IBM i(AS/400) 고객사 약 15,000개 이상 주장의 출처 확인
- [ ] 약 4,000~5,000개사가 COBOL/AS400을 사용한다는 주장의 출처 확인
- [ ] METI `2025年の崖`와 연간 최대 12조 엔 손실 표현의 원문 출처 확인
- [ ] 출처 확인 전에는 수치성 주장을 본문에 직접 쓰지 않기
- [ ] Payroll, Hitachi 등 특정 고객/벤더명은 공개 페이지에서 사용하지 않기
- [ ] "일본은 전 세계에서 유일하게..." 같은 단정 표현은 사용하지 않기

## 6. 이미지 및 자산

- [ ] 정적 페이지 Hero 이미지 또는 시각 요소 필요 여부 결정
- [ ] 이미지가 실제 AS/400/IBM i를 직접 묘사하지 않을 경우 과장된 시각 표현 피하기
- [ ] 시스템 분석, 코드 변환, 단계적 마이그레이션 흐름을 설명하는 다이어그램 필요 여부 결정
- [ ] alt 텍스트 일본어 작성
- [ ] 이미지 경로가 라우트와 정렬되는지 확인

## 7. 구현 전 확인

- [ ] 기존 일본 웹사이트 CTA URL 확인
- [ ] 공개 경로와 프리뷰 경로 정책 확인
- [ ] sitemap 포함 여부 확인
- [ ] 관련 테스트 필요 여부 확인
- [ ] 배포 전 stage 또는 preview URL에서 표시 확인
