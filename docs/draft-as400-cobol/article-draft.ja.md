# AS400 / COBOL Migration

AS/400（IBM i）とCOBOL/RPGで動く基幹システムを、AIで分析し、Java、PostgreSQL、Linux、Cloud環境への移行を段階的に進めるモダナイゼーションサービスです。

長年運用されてきた基幹システムには、受発注、在庫、会計、決済、給与、顧客管理など、企業の業務を止められない重要なロジックが蓄積されています。

その一方で、保守コストの増加、COBOL/RPGエンジニアの高齢化、システムのブラックボックス化、クラウド移行の遅れは、多くの企業にとって無視できない課題になっています。

QueryPieは、既存のAS/400（IBM i）とCOBOL/RPG資産をただ置き換えるのではなく、AIを活用して現行システムを分析し、移行範囲、影響度、設計、テスト、実装方針を整理しながら、現実的なモダナイゼーションを支援します。

## 日本企業に残るAS/400（IBM i）とCOBOL/RPG

日本では、AS/400（IBM i）やCOBOL/RPGで構築された基幹システムが、現在も多くの企業で利用されています。

特に、製造、流通、物流、金融、保険、BPO、給与計算など、止められない業務を支えるシステムでは、長年運用されてきたAS/400（IBM i）環境が今も重要な役割を担っています。

これらのシステムは、単に古いから残っているわけではありません。長期間にわたって安定稼働し、業務に深く組み込まれ、企業ごとの複雑なビジネスロジックを支えてきたからこそ、簡単に置き換えることができません。

多くの企業では、次のような判断が長く続いてきました。

> 動いているから、今は触らない。

しかし、その判断を続けるだけでは、これからの維持が難しくなりつつあります。

## なぜ今、AS400 / COBOL Modernizationが必要なのか

AS/400（IBM i）とCOBOL/RPGの課題は、システムが動かないことではありません。むしろ、安定して動いているからこそ、変える判断が遅れやすいことにあります。

一方で、運用の現場では次の課題が大きくなっています。

- COBOL/RPGを理解できるエンジニアが減っている
- 既存ベンダーや特定の保守担当者への依存が高まっている
- ハードウェア、ライセンス、保守費用が重くなっている
- 長年の改修で業務ロジックの全体像が見えにくくなっている
- 影響範囲が分からないため、改修や移行の判断が遅れる
- クラウド、API、データ活用などのDX施策と接続しにくい

日本では、経済産業省が指摘した「2025年の崖」を背景に、レガシーシステムのブラックボックス化、IT人材不足、維持コストの増加が大きな経営課題として認識されています。

AS400 / COBOL Modernizationは、単なる技術刷新ではありません。企業のコア業務を支える既存資産を活かしながら、次の10年も運用できる形に変えていく取り組みです。

## 全面再開発ではなく、まずAIで現行システムを理解する

AS/400（IBM i）やCOBOL/RPGの移行が難しい理由は、コード量だけではありません。

本当に難しいのは、長年の運用で蓄積された業務ロジック、データ構造、外部連携、例外処理、バッチ処理の関係を正確に理解することです。

QueryPieは、AIを活用して既存システムの理解から始めます。

### AIによる分析

- COBOLプログラムの構造分析
- RPGロジックの分析
- DB2 / Oracleなどのデータ構造分析
- プログラム、テーブル、バッチ、外部連携の依存関係整理
- 改修や移行時の影響度分析

### AIによるドキュメント生成

- 現行仕様書の生成
- 業務ロジックの整理
- データモデルの作成
- 移行対象の分類
- テストケースの生成

これにより、まず「何を移行すべきか」「どこから着手すべきか」「どの範囲に影響があるのか」を見える化します。

## Java / PostgreSQL / Linux / Cloudへの段階的な移行

現行システムを理解したうえで、移行先のアーキテクチャを段階的に設計します。

移行の方向性は、企業の状況に応じて異なります。

- COBOLからJavaへの変換
- RPGからJavaへの変換
- DB2 / OracleからPostgreSQLへの移行
- AS/400（IBM i）からLinux環境への移行
- OCI / AWSなどのクラウド基盤への移行
- 既存機能をAPI Wrapperで段階的に外部化
- 既存IBM iを残しながら一部機能だけを先に切り出す

重要なのは、いきなり全体を置き換えることではありません。

現行システムを分析し、影響度を把握し、業務上の優先順位を決め、小さく検証しながら段階的に移行することです。

## QueryPieの進め方

QueryPieは、AS400 / COBOL Migrationを次のステップで進めます。

### 1. 現状診断

対象システム、COBOL/RPGプログラム、データベース、バッチ、外部連携を確認し、移行の全体像を把握します。

### 2. AI分析

AIを活用して、コード構造、業務ロジック、データモデル、依存関係、影響範囲を分析します。

### 3. 移行計画

Java化、PostgreSQL化、API化、Cloud移行などの候補を整理し、段階的な移行計画を作成します。

### 4. PoC

一部のプログラムや業務領域を対象に、変換精度、テスト可能性、移行後の運用イメージを検証します。

### 5. 段階的な本番移行

業務影響を抑えながら、優先度の高い領域から順に移行を進めます。
