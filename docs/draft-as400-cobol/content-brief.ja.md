# AS400 / COBOL Migration コンテンツ企画書

## 1. コンテンツ概要

### 作業名

日本向けWebサイト用 IBM i（AS/400）モダナイゼーション 静的Webページ

### コンテンツ目的

日本企業に現在も多く残る IBM i（AS/400）と COBOL/RPG ベースの基幹システムを対象に、QueryPie AI が提供できる AI ベースのモダナイゼーションアプローチを紹介します。

中心テーマは、次の課題です。

- AS/400（IBM i）と COBOL/RPG の保守コスト増加
- COBOL/RPG エンジニアの高齢化と人材不足
- 長年蓄積されたビジネスロジックのブラックボックス化
- 既存ベンダー／SI への依存度の高まり
- クラウド移行とDXの遅れ
- Java、API、Cloud、PostgreSQL、Linux、OCI/AWS ベースのモダナイゼーション需要

### 公開形式の優先順位

一次公開形式は、日本向けWebサイトの静的Webページです。公式主表記は `IBM i（AS/400）モダナイゼーション` とし、`AS400 / COBOL マイグレーション` は SEO と文脈上の補助表記として使用します。

初期実装は `/services/as400-cobol` に直接アクセスできるコンテンツ初稿です。コンテンツは複数回のレビューと修正を前提とするため、ナビゲーションと sitemap にはまだ露出せず、robots は `noindex, nofollow` とします。

後続コンテンツとして、次の形式を検討します。

| 優先順位 | 形式 | 目的 |
|----------|------|------|
| 1 | 静的Webページ | `/services/as400-cobol` コンテンツ初稿の実装 |
| 2 | White paper | リード獲得、詳細診断フレーム、営業フォロー資料 |
| 3 | Blog | 市場教育、SEO流入、課題認知の拡大 |
| レビュー用 | 直接URL | ナビゲーション／sitemap 非露出状態で stage または production 確認 |

## 2. 対象読者

### 主な読者

- 日本の中堅・大企業の CIO、CTO、DX推進責任者
- 基幹システム、Core System、基幹システムを担当する組織
- AS/400（IBM i）、COBOL、RPG ベースのシステムを運用する情報システム部門
- 製造、流通、物流、金融、保険、BPO、Payroll 領域のシステム責任者
- 既存SI／ベンダーの保守コストと移行リスクを同時に検討する意思決定者

### 読者の課題

- AS/400（IBM i）と COBOL/RPG システムがコア業務を支えており、簡単には変更できません。
- システムは安定稼働している一方で、保守人材とコストの課題が大きくなっています。
- 長年蓄積された業務ロジックがコード内に残り、移行範囲と影響度を把握しにくくなっています。
- 既存ベンダーに高い費用を支払い続けているものの、具体的なマイグレーションロードマップを描きにくい状況です。
- 全面再開発はコストとリスクが大きいため、段階的で検証可能な移行方式が必要です。

## 3. 主要メッセージ

### メインメッセージ

IBM i（AS/400）と COBOL/RPG ベースの基幹システムは、日本企業のコア業務を支えてきました。しかし今後は、保守コスト、人材不足、ブラックボックス化、DXの遅れを解決するためのモダナイゼーションが必要です。QueryPie AI は AI を活用して既存の COBOL/RPG ロジックを分析し、Java / API / Cloud / PostgreSQL / Linux / OCI / AWS ベースの移行を段階的に進められるよう支援します。

### 補助メッセージ

1. レガシーシステムのモダナイゼーションは、全面再開発ではなく、まず現行システムを正確に理解することから始まります。
2. AI は COBOL/RPG、DB2/Oracle、影響度分析、設計書生成、テストケース生成に活用できます。
3. 分析結果をもとに、Java 変換、API 生成・連携、PostgreSQL/Linux/OCI/AWS/Cloud 移行を段階的に検討できます。
4. コア業務を止めないために、診断、PoC、変換、実装、テスト、運用安定化まで続く段階的な進め方が必要です。
5. 日本市場では `2025年の崖`、COBOL エンジニアの高齢化、IBM i 保守コストの増加がモダナイゼーション需要を高めています。

## 7. 確認が必要な事項

- `IBM i（AS/400）モダナイゼーション` を公式主表記として使用する。`AS400 / COBOL マイグレーション` は検索性と文脈のための補助表記として使用する。
- AI ベースの COBOL/RPG 分析、設計書生成、テストケース生成、Java/API/Cloud 移行は十分に公開文言として使用可能
- AS/400（IBM i）直接接続、DB2/Oracle 分析、Java 変換などの技術範囲を明確に確認
- Payroll 事例における具体的な期間感は、Brant がすでにシステム状況を大まかに把握していた特定顧客文脈のものである。一般的なホームページ公開文言では特定の期間数値を使わず、初期診断と PoC を通じて現行システムを理解し、移行範囲とロードマップを具体化する流れとして表現する。
- 相談CTAの基本パスは `/contact-us` として確定する。ボタン文言は `相談する` とし、問い合わせ目的や製品カテゴリを事前選択するための query string 設計は後続課題として残す。
- 静的Webページの初期実装パスは `/services/as400-cobol` として確定する。コンテンツレビューが完了する前でも stage/production にデプロイされる可能性があるため、今回の初稿ではナビゲーションと sitemap に露出せず、robots を `noindex, nofollow` に設定する。
- 15,000社、4,000〜5,000社、年間12兆円などの数値主張は初稿で可能な範囲で示し、出典確認は別途行う。出典検証前は断定を避け、留保表現を使用する。
- Hero 画像は `public/services/as400-cobol/` 配下の route-aligned bitmap asset とする。本文には段階的 Migration の流れを図とカード／パネルの混合型で実装する。
